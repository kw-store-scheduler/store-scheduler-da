# Part 1. 최적화 엔진

## 1. 기술 스택 및 기본 구조

### 사용 기술

| 항목 | 내용 |
|------|------|
| 언어 | Python 3.x |
| 최적화 라이브러리 | Google OR-Tools CP-SAT (`ortools.sat.python.cp_model`) |
| 백엔드 연동 | Spring Boot (REST API 호출 → `solve_schedule(config)` 반환) |

### CP-SAT 선택 이유

CP-SAT(Constraint Programming + Boolean SATisfiability)는 스케줄링처럼 **이진 결정변수**(근무함/안 함)와 **복합 제약**이 공존하는 문제에 특화된 솔버다. 선형 계획법(LP)보다 표현력이 높고, 정수 계획법(MIP) 대비 실전 스케줄 문제에서 빠른 수렴을 보인다.

### 결정 변수

```
work[e, d, s] ∈ {0, 1}
  e : 직원 인덱스
  d : 날짜 인덱스 (0-based)
  s : 교대 인덱스
```

직원 e가 날짜 d에 교대 s로 근무하면 1, 아니면 0.

### 하드 제약 (반드시 준수)

| 번호 | 내용 | 구현 방식 |
|------|------|-----------|
| 하드 1 | 가용 요일 외 근무 불가 | `model.Add(work[e,d,s] == 0)` |
| 하드 2 | 하루 최대 1교대 (이중 근무 금지) | `model.AddAtMostOne(...)` |
| 하드 3 | 주간/월간 최대 근무시간 초과 금지 | `model.Add(weekly_hours <= max_hours)` |
| 하드 4 | 연속 5일 이상 근무 금지 | 5일 슬라이딩 윈도우 합산 ≤ 4 |

## 2. 문제 1 — 인원 부족 시 스케줄 자체가 생성되지 않던 문제

### 발생 상황

교수님 피드백: "월요일에 3명이 필요한데 가용 인원이 2명뿐이면 어떻게 되나요?"

초기 코드는 교대별 최소 인원을 **하드 제약**으로 처리하고 있었다.

```python
# 구 코드 — [하드 5]
for d in range(D):
    for s in range(S):
        model.Add(sum(work[e, d, s] for e in range(E)) >= min_s[s])
```

가용 인원이 최소 인원보다 적으면 이 제약을 절대 만족시킬 수 없으므로 OR-Tools는 즉시 **INFEASIBLE**을 반환하고 스케줄을 전혀 생성하지 않았다.

### 해결 방법: 하드 → 소프트 제약 전환

`[하드 5]`를 완전히 제거하고, 대신 **소프트 제약**으로 대체했다. 위반 시 매우 높은 패널티를 부과해 가용 인원이 있으면 반드시 채우도록 유도하되, 인원 자체가 부족한 경우에도 스케줄은 생성하고 부족 현황을 보고한다.

```python
# 신 코드 — [소프트 0]
MIN_STAFF_W = 50_000_000  # 1명 부족 = 5천만 패널티

for d in range(D):
    for s in range(S):
        actual = sum(work[e, d, s] for e in range(E))
        for k in range(1, min_s[s] + 1):
            v = model.NewBoolVar(f"ms_{d}_{s}_{k}")
            model.Add(actual <= k - 1).OnlyEnforceIf(v)   # v=1 ↔ actual < k
            model.Add(actual >= k).OnlyEnforceIf(v.Not())
            min_shortage_vars.append(v)
```

### 결과

| 상황 | 구 버전 | 신 버전 |
|------|---------|---------|
| 인원 충분 | OPTIMAL + 스케줄 | OPTIMAL + 스케줄 |
| 인원 부족 | **INFEASIBLE + 스케줄 없음** | **OPTIMAL + 스케줄 + shortage_report** |

## 3. 문제 2 — 복수 최적해 감지 및 처리

### 발생 상황

교수님 피드백: "최적해가 여러 개인 경우 어떻게 처리할 것인가?"

CP-SAT는 최적해를 **하나만** 반환한다. 동일한 목적함수 값을 가진 다른 배정 방식이 존재하는지 알 수 없었다.

### 해결 방법: No-Good Cut 기법

첫 번째 최적해를 찾은 뒤, 그 해를 **배제하는 절(No-Good Cut)**을 모델에 추가하고 재풀기를 반복한다. 재풀기 결과의 목적함수 값이 첫 번째와 동일하면 또 다른 최적해가 존재하는 것이다.

```python
def _add_no_good_cut(model, work, solver, E, D, S):
    literals = []
    for e in range(E):
        for d in range(D):
            for s in range(S):
                if solver.Value(work[e, d, s]) == 1:
                    literals.append(work[e, d, s].Not())
                else:
                    literals.append(work[e, d, s])
    model.AddBoolOr(literals)  # "이 해와 완전히 같지는 않아야 한다"
```

### 복수 해 탐색 흐름

```
1차 풀기 → OPTIMAL 확인 → 목적함수 값 저장
    ↓
No-Good Cut 추가 → 2차 풀기
    ↓
목적함수 값 동일? → has_multiple_optimal = True, alt_schedules에 추가
    ↓
최대 (max_alt_solutions - 1)회 반복
```

## 4. 문제 3 — 타이(동등 최적해) 감소 전략

### 타이가 생기는 근본 원인

`target_staff = [2, 2]`처럼 오픈·마감을 동등하게 설정하면 **대칭성**이 발생한다. 예: 화요일에 배정 가능한 직원이 3명인데, 누가 오픈/마감을 맡아도 UNDERSTAFF 패널티가 동일 → 솔버 입장에서 완전히 동등한 해.

이 타이는 구조적 원인이므로 완전히 없애기보다 **최대한 줄이는 전략**을 사용했다.

### 4-1. 휴무 요청 (off_requests) — 2차 수정에서 추가

```python
# employee에 추가 (선택)
"off_requests": [
    {"day": 4, "type": "must"},    # 필수 휴무 — 거의 하드처럼 작동
    {"day": 6, "type": "prefer"},  # 희망 휴무 — 가능하면 배정 안 함
]
```

직원마다 고유한 휴무 요청이 있으면 해 공간이 비대칭해져 많은 타이가 자연히 사라진다. 위반 시 `off_request_violations` 필드에 기록된다.

| 타입 | 패널티 | 동작 |
|------|--------|------|
| `must` | 10,000,000 / 건 | 사실상 하드 제약. 인원 부족이 아니면 100% 준수. |
| `prefer` | 200,000 / 건 | 최대한 준수하되 불가피하면 위반 후 보고. |

### 4-2. 근무일수 균등 배분 (FAIRNESS_W) — 2차 수정에서 추가

직원 간 근무일수 최대-최소 편차를 최소화해 롤링 배분을 유도한다.

```python
FAIRNESS_W = 10  # 타이브레이킹용 미세 조정 (주 목적함수에 영향 없음)

model.AddMaxEquality(max_wd, work_days_vars)
model.AddMinEquality(min_wd, work_days_vars)
# 패널티 += FAIRNESS_W * (max_wd - min_wd)
```

**효과**: W08 완전 스케쥴제 시나리오에서 8명 전원이 고정 휴무 없이도 서로 다른 날 쉬는 롤링 스케줄이 자동 생성됨.

### 4-3. 공휴일 근무 균등 배분 (HOLIDAY_W) — 2차 수정에서 추가

```python
"holiday_indices": [5]    # config에 추가 (선택) — 0-based 날짜 인덱스

HOLIDAY_W = 5  # FAIRNESS_W보다 낮은 가중치
```

공휴일 근무 횟수의 직원 간 편차를 최소화한다.

### 4-4. 역할 제약 (role_level) — 설계 확정, 구현 예정

**설계 변경 이력:**

초기 설계: 숫자가 높을수록 고직급 (점장=4, 사원=1)

```python
# 초기 설계 — 폐기
"shift_requirements": [{"shift": 0, "min_level": 3, "count": 1}]
# 문제: "레벨 3 이상"은 한국어 직관과 반대 방향 (낮은 수가 높은 직급으로 읽힘)
```

**변경 이유:**
1. 한국어 맥락에서 "1등 > 2등"처럼 낮은 숫자가 높은 직급을 의미하는 것이 자연스러움
2. 제약 방향이 `max_level <= X` 형태여야 "레벨 2 이하인 직원"이라는 문장과 일치
3. 매장마다 직급 이름이 다르고(점장/팀장/원장 등) 단계 수도 다양하므로, 이름이 아닌 숫자만으로 처리해야 범용성이 생김
4. 관리자(role_level=1)가 스케줄에 참여하지 않는 경우도 있으므로 `available_days=[]`로 처리, 별도 필드 불필요

**최종 확정 설계:**

```python
# employee에 추가 (선택)
"role_level": 1   # 1=최상위(점장/매니져), 숫자 클수록 낮은 직급
                  # 스케줄 미참여 관리자: available_days=[] 로 표현

# config에 추가 (선택)
"shift_requirements": [
    {"shift": 0, "max_level": 2, "count": 1}
    # → 오픈(shift 0)에 role_level ≤ 2인 직원이 최소 1명
]
```

| 매장 유형 | 직급 구성 예시 |
|----------|-------------|
| 편의점 | 점장(1) → 대리(2) → 주임(3) → 사원(4) |
| 카페 | 매니져(1) → 바리스타(2) → 파트(3) |
| 미용실 | 원장(1) → 디자이너(2) → 인턴(3) |

## 5. 패널티 가중치 계층 설계

```
목적함수 = 인건비 + 야간수당
         + MIN_STAFF_W   × 최소인원_부족_건수       ← 최우선
         + MUST_OFF_W    × 필수휴무_위반_건수        ← 2차 수정 추가
         + UNDERSTAFF_W  × 목표인원_미달_건수
         + LOW_HOURS_W   × 주15h미만_직원수
         + PREFER_OFF_W  × 희망휴무_위반_건수        ← 2차 수정 추가
         + PREF_W        × 비선호교대_배정_건수
         + FAIRNESS_W    × 근무일수_최대최소_편차    ← 2차 수정 추가 (타이브레이킹)
         + HOLIDAY_W     × 공휴일근무_최대최소_편차  ← 2차 수정 추가 (타이브레이킹)
```

| 상수 | 값 | 의미 |
|------|-----|------|
| `MIN_STAFF_W` | 50,000,000 | 최소 인원 1명 부족 |
| `MUST_OFF_W` | 10,000,000 | 필수 휴무 1건 위반 |
| `UNDERSTAFF_W` | 5,000,000 | 목표 인원 1명 미달 |
| `LOW_HOURS_W` | 2,000,000 | 주 15시간 미만 직원 1명 |
| `PREFER_OFF_W` | 200,000 | 희망 휴무 1건 위반 |
| `PREF_W` | 100,000 | 비선호 교대 배정 1건 |
| `FAIRNESS_W` | 10 | 근무일수 편차 (타이브레이킹) |
| `HOLIDAY_W` | 5 | 공휴일 편차 (타이브레이킹) |

**설계 원칙**: 상위 항목 1건 위반이 하위 항목 수백 건 위반보다 비싸도록 설정해 우선순위가 확실히 분리된다.
